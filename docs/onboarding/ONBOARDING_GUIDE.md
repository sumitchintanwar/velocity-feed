# RTMDS Engineering Onboarding Guide

Welcome to the Real-Time Market Data System (RTMDS) team! This guide is designed to accelerate your ramp-up time so you can begin contributing to our high-throughput distributed platform confidently.

## 1. Project Overview

**The Mission:** RTMDS is a low-latency, highly scalable WebSocket gateway designed to ingest normalized market data from various upstream exchanges (via feed providers), buffer it, and fan it out to thousands of connected clients with sub-millisecond tail latency.

**The Architecture:** 
The platform is written in Go and utilizes a distributed topic routing architecture. 
- **Gateways** manage client WebSocket connections and internal ring buffers.
- **Publishers** normalize and publish incoming market data ticks.
- **Redis Pub/Sub** provides cross-node message routing (ensuring a client connected to Gateway A can receive a tick generated by Publisher B).
- **PostgreSQL** stores historical data for time-travel replay.

## 5. Frequently Asked Questions (FAQ)

**Q: Why do we use Redis Pub/Sub instead of Kafka?**
A: RTMDS optimizes for low-latency *volatile* messaging. Kafka is optimized for persistent log streaming. Redis Pub/Sub provides significantly lower latencies for instantaneous fan-out, which is critical for real-time market data. We use PostgreSQL separately for durable historical replay.

**Q: What happens if a WebSocket client is too slow?**
A: RTMDS employs dynamic backpressure. Each client has a bounded ring buffer. If a client falls behind and the buffer fills, the Gateway will proactively drop the oldest messages to maintain system stability, incrementing a `dropped_messages` counter.

**Q: How do I run the performance tests?**
A: Use the provided `make stress-test` command which automates progressive load injection via `cmd/benchmark` to locate the system saturation point.
